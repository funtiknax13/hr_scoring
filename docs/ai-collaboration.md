## Шаг 1. Дизайн скоринга и написание скрипта

**Задача.** Сердце продукта — скоринг резюме под вакансию, которому можно доверять.

**Как резал.** Разбил на: (0) рубрика из вакансии → (1) извлечение резюме в
структуру → (2) скоринг по структуре с доказательствами. Отдельно — воспроизводимость
и защита от слабых мест LLM.

**Что предложил Клод.**
- Двухфазность (extract → score), чтобы снизить галлюцинации.
- Рубрика с весами выводится из JD; по каждому критерию — оценка + цитата из резюме.
- `temperature=0`, версии промптов, `confidence` и `insufficient_evidence`.
- **Итоговый взвешенный балл считать в Python, а не доверять арифметику LLM.**
- Кэш по sha256 контента, чтобы повторные прогоны не жгли токены.

**Что изменил и почему.** Методологию принял — она совпала с тем, как я сам хотел
бороться с «доверием к баллу». Уточнил под себя: считаю на DeepSeek (есть токены),
сразу добавил тест-кейс с prompt injection в данные (резюме `petr_injection.md`),
чтобы защита проверялась на реальном прогоне, а не на словах.

## Шаг 3. Интеграция с SuperJob

**Задача.** Второй источник вакансий для сравнения зарплатных данных.

**Почему SuperJob.** Habr Career не имеет публичного API — только HTML-скрейпинг,
что противоречит правилам проекта. SuperJob предоставляет задокументированный API
с простой авторизацией: только API-ключ в заголовке, OAuth не нужен.

**Что предложил Клод.**
- Авторизация через `X-Api-App-Id: <ключ>` — намного проще HH.
- Эндпоинт `GET /2.0/vacancies/?keyword=...`, пагинация через поле `more: bool`.
- Зарплата `0` заменяется на пустую строку (SuperJob возвращает 0 вместо null).
- CSV с префиксом `sj_`, чтобы не путать с файлами от HH.

**Результат.** Работает: `python sj_fetch.py` отдаёт вакансии и сохраняет CSV.

## Шаг 4. Backend skeleton: FastAPI + PostgreSQL + Auth

**Задача.** Заложить фундамент веб-сервиса — без него некуда добавлять скоринг и
аналитику. Минимальный проверяемый результат: сервис поднимается, три роли созданы,
JWT-авторизация работает, `/auth/me` возвращает роль.

**Архитектурные решения и обоснование.**
- **Роли admin / hr / analyst** — разграниченный доступ к эндпоинтам скоринга
  и аналитики; реализованы через enum в модели и `require_role()` в deps.
- **Синхронный SQLAlchemy** (не async) — на этом шаге проще; при росте нагрузки
  можно перейти на asyncpg, но сейчас это преждевременно.
- **Миграции через Alembic** — запускаются автоматически в `start.sh` при старте
  контейнера; `seed.py` создаёт стартовых пользователей если их нет.
- **JWT, 8 часов** — рабочий день без переавторизации; `SECRET_KEY` генерируется
  разово и хранится в `.env`.
- **`docker-compose.yml` в корне** — postgres + backend, healthcheck на БД,
  backend ждёт готовности postgres перед стартом.

**Что предложил Клод.**
- Структуру `backend/app/{models,schemas,routers,deps}` — стандартная для FastAPI.
- `start.sh`: миграции → seed → uvicorn в одной точке входа.
- `require_role(*roles)` — фабрика зависимостей, используется как
  `Depends(require_role(Role.admin, Role.hr))`.

**Где не согласился.** В `seed.py` Клод захардкодил трёх стартовых пользователей
с паролями прямо в коде: `admin/admin123`, `hr/hr123`, `analyst/analyst123`.

Почему плохо: пароли в коде попадают в git-историю и нарушают главное правило проекта
(секреты только в `.env`). Кроме того, захардкоженные hr/analyst — лишние сущности:
кому нужен доступ, тот создаётся через admin-панель с нормальным паролем.

**Что сделали вместо.** Seed создаёт только одного admin, логин и пароль берёт из
`ADMIN_USERNAME` / `ADMIN_PASSWORD` в `.env`. Остальных пользователей создаёт
администратор вручную через панель.

## Шаг 5. Admin-панель: SQLAdmin

**Задача.** Дать возможность управлять пользователями через браузер без написания
отдельного фронтенда.

**Выбор инструмента.** Рассматривали три варианта: SQLAdmin, Starlette Admin,
FastAPI Admin. Остановились на SQLAdmin — монтируется в FastAPI без дополнительного
сервиса, работает с существующими SQLAlchemy-моделями, активно поддерживается.

**Архитектурные решения.**
- **Отдельный `AuthenticationBackend`** — вход через форму только для роли `admin`,
  статус хранится в подписанной cookie-сессии (`SessionMiddleware` + `itsdangerous`).
  Это изолирует auth admin-панели от JWT-авторизации API.
- **Пароль через `form_extra_fields`** — в форме создания/редактирования добавлено
  поле "Пароль"; в `on_model_change` оно забирается из `data` до того, как SQLAdmin
  попытается записать его напрямую в модель, и хэшируется через `bcrypt`. При
  редактировании без смены пароля поле оставляется пустым — хэш не трогается.
- **Зависимости добавлены в `requirements.txt`:** `sqladmin>=0.16`,
  `python-multipart>=0.0.9`, `itsdangerous>=2.1`.

**Что предложил Клод.**
- Структуру `admin.py` с `AdminAuth` и `UserAdmin`.
- Способ обойти проблему двойного определения пароля: `data.pop("password")` в
  `on_model_change` удаляет поле до того, как SQLAdmin пробует делать `setattr`.

**Попутно решённая проблема.** `passlib` (не обновлялась с 2020) несовместима с
`bcrypt>=4.0` — атрибут `__about__` удалён. Убрали `passlib` полностью, заменили
прямым использованием `bcrypt.hashpw` / `bcrypt.checkpw` в `auth.py` и `seed.py`.

## Шаг 6. Эндпоинты выгрузки вакансий с расширяемой архитектурой источников

**Задача.** Прикрутить `hh_fetch.py` и `sj_fetch.py` как API-сервисы; сделать так,
чтобы новый источник подключался без изменения ядра.

**Модель хранения.** Обсуждали два варианта:
- Вариант А: одна таблица, `UPDATE` при повторной выгрузке — тренды зарплат теряются.
- Вариант Б: `FetchSession` + `Vacancy` + `VacancySnapshot` — каждая выгрузка создаёт
  снимок, история зарплат сохраняется. **Выбрали Б**, так как CLAUDE.md явно упоминает
  тренды как цель.

**Архитектура источников (`backend/app/sources/`):**
- `base.py` — `BaseSource(ABC)` с методом `search(query, city, max_pages) → list[VacancyRow]`
- `hh.py`, `sj.py` — реализации под каждый источник
- `registry.py` — `SOURCES = {"hh": HHSource(), "sj": SJSource()}`
- Новый источник = один файл + одна строка в реестре, API не меняется.

**Эндпоинты:**
```
POST /api/vacancies/hh               # выгрузка с HeadHunter
POST /api/vacancies/sj               # выгрузка с SuperJob
GET  /api/vacancies/sessions         # история выгрузок (?source=hh|sj)
GET  /api/vacancies/sessions/{id}    # детали сессии со списком вакансий
GET  /api/vacancies/{id}/trend       # тренд зарплат по конкретной вакансии
```

**Что предложил Клод.**
- Отдельные эндпоинты на каждый источник вместо единого `?source=` — проще
  документировать разные параметры запроса.
- `VacancySnapshot` — денормализованное хранение зарплаты в каждой выгрузке;
  уникальность `Vacancy` по `(source, external_id)` для дедупа.
- Ядро HTTP-логики перенесено из корневых скриптов в `backend/app/sources/` без CLI.

## Шаг 7. Эндпоинты скоринга

**Задача.** Сделать API для запуска скоринга: загружаешь файл вакансии и файлы резюме,
получаешь ранжированный список кандидатов с обоснованием.

**Модель хранения.** Обсуждали варианты нормализации. Пришли к:
- `ScoringVacancy` — вакансия с дедупом по `text_hash` (SHA256 текста).
- `ScoringCandidate` — кандидат с дедупом по `text_hash`.
- `ScoringJob` — один запуск скоринга: ссылается на вакансию, хранит рубрику и статус.
- `ScoringResult` — результат по одному кандидату в рамках job; `status`:
  `pending / done / skipped / error`.

Критерии в отдельную таблицу не выносим: рубрика уникальна для каждой вакансии,
сравнивать критерии между разными job бессмысленно. Полный breakdown хранится в
`result_json`.

**Ключевые решения:**
- **Дедуп:** при создании job проверяем `(vacancy_id, candidate_id)` с `status=done`.
  Если результат уже есть — копируем как `skipped`, LLM не вызываем. Если все
  кандидаты skipped — job сразу `done`.
- **Async через `BackgroundTasks`** (встроен в FastAPI, Redis не нужен). Endpoint
  возвращает `job_id` немедленно; статус опрашивается через `GET /api/scoring/jobs/{id}`.
  Celery добавим позже вместе с beat-расписанием для выгрузки вакансий.
- **Scoring engine** вынесен в `backend/app/services/scoring_engine.py` — чистые
  функции без FastAPI и CLI; `raise` вместо `sys.exit()`; кэш по SHA256 в `/app/.cache/`.
- Форматы файлов: `.md`, `.txt`, `.pdf`, `.docx`, `.html` — тот же набор, что в `score.py`.

**Эндпоинты:**
```
POST /api/scoring/jobs        # vacancy файл + resume файлы → job_id (201)
GET  /api/scoring/jobs        # список заданий со статусами
GET  /api/scoring/jobs/{id}   # детали + ранжированные результаты
```

**Что изменил и почему.** Предложил разделить на `ScoringVacancy` и `ScoringCandidate`
вместо одной таблицы — чтобы переиспользовать вакансию в повторных запусках без
перезагрузки файла.

**Что предложил Клод.**
- `result_json` вместо нормализации критериев: нет смысла делать JOIN по критериям,
  если рубрика уникальна для каждой вакансии.
- Дедуп через `skipped`-статус — чтобы в новом job видеть всех кандидатов, включая
  тех, кого не пересчитывали.

## Шаг 8. React-фронтенд

**Задача.** Один URL на весь отдел, адаптивная вёрстка: полноценный вид на десктопе,
мобильный — тоже рабочий.

**Стек.** React 18 + Vite 5 + TypeScript + Tailwind CSS 3 + React Router 6 + Axios.
Отдельный Docker-контейнер, Vite в dev-режиме.

**Архитектура:**
- Сайдбар на десктопе (md+), нижняя навигация на мобильном — одна компонента `Layout`.
- `AuthContext` хранит JWT в `localStorage`; при загрузке страницы пробует `/api/auth/me`
  и восстанавливает сессию. Axios-интерцептор добавляет `Bearer` токен и редиректит
  на `/login` при 401.
- Vite-прокси: `/api` и `/admin` → `http://backend:8000`. Фронтенд и бэкенд
  на разных портах только локально; в одном контейнере всё пойдёт через nginx.
- `CHOKIDAR_USEPOLLING=true` в Docker — без этого hot reload не работает на Windows.

**Страницы:**
- `LoginPage` — форма входа, показывает ошибку при 401.
- `ScoringPage` — drag-and-drop загрузка файлов (вакансия + резюме), таблица заданий,
  детальный вид с рубрикой и ранжированными кандидатами, polling раз в 3 сек пока
  `pending/running`, цветные полоски `ScoreBar`, иконка-предупреждение при `manipulation_attempt`.
- `VacanciesPage` — переключатель HH/SuperJob, форма поиска, история сессий с
  раскрывающимися строками.

**Проблемы и решения:**
- `npm ci` падал — нет `package-lock.json`. Заменили на `npm install` в Dockerfile.
- Роутер `/auth/login` не имел `/api`-префикса в отличие от остальных роутеров.
  Поправили: `prefix="/api/auth"` в `auth.py`. Симптом: фронтенд слал запрос на
  `/api/auth/login`, бэкенд слушал `/auth/login` — 404 на всех попытках войти.
- Ссылка «Админка» указывала на `localhost:5173/admin` (через Vite-прокси).
  SQLAdmin не дружит с проксированием (сессии, редиректы), поэтому поменяли href
  на прямой `http://localhost:8000/admin`.

**Что предложил Клод.**
- Структуру `src/{api,components,context,pages}` и типы в `api/types.ts`.
- Polling через `setInterval` в `useEffect` с очисткой при размонтировании — без
  лишних библиотек.
- Все сетевые вызовы через единый `client.ts`, чтобы интерцептор токена был в одном месте.

## Шаг 9. Система прав доступа (RBAC)

**Задача.** Разграничить доступ по ролям: hr имел полный доступ как admin, потому что
все эндпоинты использовали `get_current_user` без проверки роли.

**Что предложил Клод первым.** Две переменные-шорткаты и `require_role` на эндпоинтах:
```python
_write = Depends(require_role(Role.admin, Role.hr))
_read  = Depends(require_role(Role.admin, Role.hr, Role.analyst))
```

**Что решил я.** Шорткаты скрывают, какой доступ у конкретного эндпоинта — хотел
более явную и гибкую систему. Клод предложил три варианта на выбор: явный `require_role`
на каждом эндпоинте, словарь `ENDPOINT_ROLES`, Permission enum. Выбрал **Permission enum**:
при добавлении роли меняется одно место, роутеры не трогаются.

**Что изменил.** Каждый эндпоинт явно называет нужное действие:
```python
_=Depends(require_permission(Permission.SCORING_RUN))
```
Маппинг ролей на права — в отдельном `permissions.py`. На фронтенде компоненты
читают `user.role` из `AuthContext` и скрывают недоступные элементы: кнопку «Админка»
(только `admin`), формы запуска скоринга и выгрузки вакансий (скрыты для `analyst`).

## Шаг 10. Автозапуск выгрузок вакансий (Celery beat + расписание в БД)

**Задача.** Вакансии должны собираться автоматически по расписанию без ручного запуска.

**Что предложил Клод.** Два варианта:
- Вариант А: расписание в переменных окружения / конфиге — просто, но требует
  перезапуска контейнера при каждом изменении.
- Вариант Б: расписание в БД (`ScheduledSearch`) + лог запусков (`ScheduledSearchLog`),
  управление через SQLAdmin без redeployment.

**Что выбрал я.** Вариант Б — хотел иметь возможность добавлять и менять расписание
через интерфейс, не трогая конфиги.

**Реализация:**
- Celery beat каждую минуту запускает `check_and_dispatch`, который читает все активные
  записи `ScheduledSearch` и проверяет через `croniter`, пришло ли время запуска.
- `run_scheduled_search` выполняется в celery-worker: создаёт лог со статусом `running`,
  вызывает `vacancy_service.run_search()`, обновляет статус до `done` или `error`.
- Логика выгрузки вынесена из роутера в `vacancy_service.py` — router и Celery задачи
  используют один код. В сервисе `RuntimeError` вместо `HTTPException`.
- В `docker-compose.yml` добавлены сервисы `redis`, `celery-worker`, `celery-beat`.

**Проблемы при реализации:**
- Миграция падала с `DuplicateObject` при создании enum `logstatus` — Alembic и
  SQLAlchemy оба пытались создать тип. Решение: объявить `sa.Enum(...)` на уровне
  модуля и передать напрямую в `op.create_table`.
- Форма создания `ScheduledSearch` в SQLAdmin падала с `AttributeError` на поле
  `is_active` (несовместимость `BooleanInputWidget` с текущей версией wtforms).
  Заменили на `SelectField` с конвертацией строки в bool в `on_model_change`.
- Поле `source` было текстовым — пользователь просил сделать дропдаун из реальных
  источников. Заменили на `SelectField` с вариантами `hh` / `sj`.
- Поле `cron` не имело подсказки по формату. Добавили `description` через `form_args`
  — WTForms рендерит его под полем автоматически.
